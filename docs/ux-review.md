# 用户体验 Review

## 结论

第一版的核心闭环已经成立：用户可以收藏岗位、沉淀技能点、维护项目经验、记录面试，并根据面试问题反向补齐技能。

当前最需要继续打磨的不是采集能力，而是“下一步该做什么”的引导。现在系统已经能存很多信息，但还需要更主动地告诉用户：哪些岗位该推进、哪些技能该补、哪些项目经验还缺证据。

## 已经比较顺的地方

- 采集方式安全：用户手动打开岗位，扩展只读取当前页面，避免触发平台风控。
- 数据长期保存：SQLite 适合个人本地使用，也方便用 Navicat 查看。
- 岗位详情页已经成为一个工作台：能看职位描述、匹配度，也能记录面试。
- 面试记录和技能点打通后，系统开始从“岗位收藏器”升级为“求职复盘工具”。

## 主要体验问题

### 1. 首页还不够行动导向

现在首页偏统计。下一步可以加入：

- 最近收藏但未处理的岗位。
- 最近面试后低分技能。
- 本周建议补齐的技能。
- 需要补项目证据的技能。

目标是用户一打开后台就知道今天该看什么。

### 2. 技能库还缺“技能缺口”视角

技能库现在能看到技能列表和详情，但还不够回答：

- 这个技能为什么重要？
- 它来自哪些高匹配岗位？
- 面试里有没有答差？
- 有没有项目能证明？

建议下一版在技能详情页增加“岗位要求 / 面试反馈 / 项目证据 / 学习计划”四块。

### 3. 面试记录需要详情页

当前面试记录列表能展示问题摘要，但如果一场面试问题很多，列表会变长。建议增加面试详情页：

- 一场面试的完整问题列表。
- 每个问题的技能标签。
- 回答表现和复盘。
- 后续改进动作。

### 4. 岗位状态流转可以更产品化

当前状态是文本下拉。下一步可以做成固定流程：

```text
收藏 -> 待投递 -> 已投递 -> 面试中 -> 已面试 -> Offer / 淘汰 / 放弃
```

这样首页统计、列表筛选和面试记录会更统一。

### 5. 配置页和技能库之间还没有反馈

匹配规则调整后，旧岗位需要重新收藏才刷新评分。后续可以加一个“重新计算所有岗位匹配度”按钮。

### 6. 空状态还可以更友好

新用户第一次打开后台时，应该明确提示：

- 先启动本地服务。
- 再加载扩展。
- 到 Boss 打开岗位详情。
- 点击收藏。

这比单纯显示空列表更适合开箱即用。

## 下一轮建议优先级

### P0：发布前体验

- README 和用户指南保持中文、步骤清晰。
- 启动脚本只安装必要依赖。
- 删除不再推荐的 Playwright 自动化入口。
- GitHub Actions 自动跑测试。

### P1：闭环增强

- 首页增加“待行动”模块。
- 技能详情页增加面试反馈区。
- 增加面试详情页。
- 增加“重新计算所有岗位匹配度”。

### P2：能力成长

- 技能分层和技能地图。
- 学习资料收藏。
- 项目经验和技能证据矩阵。

### 暂不建议

- 批量自动打开岗位详情。
- 自动投递。
- 账号登录自动化。

这些功能会让体验看似更自动，但风险和维护成本都高，不符合当前产品定位。
